## 2026-06-21

  - fix: version matter-server épinglée à 1.1.0 (matter.js 0.17.3) pour éviter un downgrade lors de la réinstallation des dépendances — la version 1.0.0 rejetait certains appareils (ex: Nuki) à cause d'une vérification d'attestation plus stricte
  - fix: synchronisation du dataset Thread depuis un OTBR distant (SLZB) — le SLZB renvoie un JSON structuré au lieu d'un TLV hex ; le plugin reconstruit désormais le TLV correctement depuis les champs `NetworkKey`, `Channel`, `ExtPanId`, etc.

  daemon.js / matter.class.php / configuration.php — BLE Proxy
  - feat: ajout du mode **BLE Proxy** pour le commissioning Wi-Fi via un client Bluetooth distant
    - nouvelle option de configuration "BLE Proxy (distant)" dans la section Commissioning
    - quand activé, le serveur Matter expose `ws://<ip>:5580/ble` (protocole BLE Proxy WebSocket v1)
    - un client distant (Home Assistant, script Python/Bleak sur Windows…) se connecte sur ce endpoint
      et proxy les opérations BLE (scan, connect, GATT, BTP) vers le serveur
    - incompatible avec l'adaptateur Bluetooth local (mutuellement exclusif)
    - s'appuie sur `@matter-server/ble-proxy` (déjà inclus dans les dépendances)

  matter.class.php — getHealthReport()
  - feat: check IPv6 éclaté en 8 sous-vérifications distinctes (était un seul check disable_ipv6)
    - ipv6_kernel — IPv6 noyau global (/proc/sys/.../all/disable_ipv6)
    - ipv6_iface — IPv6 sur l'interface Matter auto-détectée (route par défaut)
    - ipv6_linklocal — présence d'une adresse fe80:: (minimum requis)
    - ipv6_global — présence d'une adresse globale/ULA — bloquant si absente (commissioning Thread impossible)
    - ipv6_forwarding — forwarding IPv6 actif = warning (mode routeur)
    - ipv6_accept_ra — warning uniquement si accept_ra=0 ET aucune adresse globale/ULA (évite faux positif NM)
    - ipv6_nmcli — bloquant si absent ou interface non-gérée, avec commandes de fix
    - ipv6_thread_routes — routes ULA Thread/RIO détectées (informatif)

## 2026-05-24

- Mise à jour des dépendances daemon : `matter-server` et `@matter-server/ws-client` `0.5.10` → `0.7.1`.
- changement de canal Thread en mode Router/leader : quand OTBR n'est pas Leader (ex: un HomePod ou autre border router détient le rôle de Leader), `dataset commit pending` restait local et n'était jamais propagé au réseau. Le changement de canal depuis l'interface ne fonctionnait pas dans ce cas. Corrigé en utilisant `dataset updater start` qui envoie un `MGMT_PENDING_SET.req` directement au Leader, avec un délai minimum de 300 secondes pour laisser le temps à tous les nœuds Thread de migrer.
- **Gestion du canal Thread dans la configuration** : nouvelle section permettant de lire le canal actuel, de le modifier avec confirmation, et de sauvegarder/restaurer le dataset avant modification.
- correction du nom de propriété pour la température de couleur .

## 2026-04-27

- **Watchdog OTBR — amélioration des logs de debug** : les cycles de retry périodique, les périodes de cooldown, les transitions `agent_down` et les redémarrages OTBR sont maintenant tracés explicitement dans les logs `matter_otbr`.
- **Watchdog OTBR — gestion automatique du redémarrage** : le watchdog détecte et gère l'état `agent_down` avec escalade progressive (redémarrage du service otbr-agent, puis du conteneur complet si nécessaire).
- Le watchdog redémarre automatiquement le conteneur OTBR s'il s'arrête de façon inattendue (cooldown de 60 secondes entre deux tentatives).
- Si le conteneur est complètement disparu ou dans un état irrécupérable, le watchdog recrée entièrement le conteneur OTBR avec la même configuration.
- Clarification des rôles : le **watchdog** gère la surveillance et le redémarrage d'OTBR, le **cron Jeedom** vérifie uniquement que le watchdog tourne et le relance si nécessaire.

:::note
Après une recréation complète du conteneur par le watchdog, le réseau Thread redémarre sans restauration automatique du dataset. Si les appareils Thread ne se reconnectent pas, utilisez **Gestion Matter & Thread → Injecter le dataset** pour restaurer votre réseau.
:::
